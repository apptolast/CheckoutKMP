01

Qué demuestra este proyecto

Simula el checkout y pago de un retail (tipo Zara/Inditex): carrito → método de pago → tarjeta → 3D Secure → resultado. No es banca core; es exactamente el flujo de cliente que se hace en pagos de e‑commerce. Cada pieza está pensada para que puedas decir “esto lo he montado”:

  • Tokenización de tarjeta PCI‑safe: el número (PAN) nunca se loguea ni se persiste en claro.
  • 3D Secure / SCA: challenge de autenticación reforzada, obligatorio por PSD2 en Europa.
  • Idempotencia: una misma clave garantiza que un reintento no cobre dos veces.
  • Máquina de estados del pago y manejo de errores / reintentos con criterio.
  • Accesibilidad en formularios de pago — tu sello: traversalIndex, liveRegion.
02

Conceptos de pagos

Lo que hay que estudiar no es Kotlin (eso lo dominas), es el vocabulario y los flujos. Con estos seis vas sobrado para una entrevista de cliente.

PCI DSS

Por qué el PAN no se loguea, no se persiste en claro y se tokeniza.

PSD2 / SCA

Autenticación reforzada del cliente. En cliente se traduce en el flujo 3D Secure 2.

PSP / gateway

El intermediario (Adyen, Stripe, Redsys…) al que se le pide autorizar el pago.

Tokenización

Cambiar los datos sensibles de la tarjeta por un token opaco reutilizable.

Idempotencia

Misma clave → mismo resultado aunque reintentes. Evita cobros dobles.

Máquina de estados

Iniciado → autenticando (3DS) → autorizado → capturado → ok / fallo / reembolso.

03

Arquitectura KMP

La lógica va a commonMain (compartida); la UI a Android. La suite de tests en commonTest valida la lógica una sola vez para todos los targets.

estructura:shared            // módulo KMP con la lógica (android; iOS opcional)
  commonMain
    domain/         // modelos + casos de uso (Kotlin puro) — el corazón
    data/           // PaymentRepository, FakePsp, tokenizer, mappers
  commonTest        // tests de la lógica (corren en todos los targets)
  androidMain       // actual específicos de Android, si hicieran falta
  iosMain           // (opcional, más adelante)
:androidApp         // UI Compose (MVI), navegación, wiring de DI
(:iosApp)           // (opcional) SwiftUI o Compose Multiplatform

Reglas: domain no depende de data; data depende de domain; la UI depende de :shared. DI con Koin. Patrón de UI: MVI (estado inmutable + intents).

Detalles propios de KMP (apréndetelos, suman)

  • UUID para las idempotency keys: kotlin.uuid.Uuid (stdlib), nada de java.util.UUID en común.
  • Dispatchers: no uses Dispatchers.Main/IO directos en común; inyéctalos para poder testear.
  • Fechas (caducidad de tarjeta): kotlinx-datetime, no java.time.
  • Tests: kotlinx-coroutines-test y Turbine funcionan en commonTest.

kotlinsealed interface PaymentState {
    data object Idle : PaymentState
    data object Processing : PaymentState                // enviado al PSP
    data class RequiresSca(val challenge: ScaChallenge) : PaymentState  // 3DS
    data object Authenticating : PaymentState            // completando 3DS

}

Caso de uso principal (puro y testable)

kotlinclass ProcessPaymentUseCase(private val repo: PaymentRepository) {
    operator fun invoke(request: PaymentRequest): Flow<PaymentState> = flow {
        emit(PaymentState.Processing)
        when (val r = repo.authorize(request)) {
            is AuthResult.Approved -> emit(PaymentState.Succeeded(r.receipt))
            is AuthResult.Declined -> emit(PaymentState.Failed(PaymentError.Declined))
            is AuthResult.NeedsSca -> emit(PaymentState.RequiresSca(r.challenge))
            is AuthResult.Error    -> emit(PaymentState.Failed(r.error))
        }
    }
}

Validación de tarjeta (Luhn)

kotlinfun isValidLuhn(pan: String): Boolean {
    val digits = pan.filter(Char::isDigit)
    if (digits.length < 12) return false
    var sum = 0; var alt = false
    for (i in digits.indices.reversed()) {
        var d = digits[i].digitToInt()
        if (alt) { d *= 2; if (d > 9) d -= 9 }
        sum += d; alt = !alt
    }
    return sum % 10 == 0
}

05

Manejo de errores y resiliencia

En pagos el camino feliz es el fácil; lo que te evalúan es cómo tratas los fallos.

Taxonomía de errores

kotlinsealed interface PaymentError {
    data object Declined : PaymentError        // rechazo del banco -> NO reintentar
    data object InvalidCard : PaymentError     // datos inválidos
    data object Network : PaymentError         // sin conexión -> reintentable
    data object Timeout : PaymentError         // el PSP no respondió -> reintentable
    data object RateLimited : PaymentError     // 429 -> backoff
    data object ScaFailed : PaymentError       // 3DS fallido
    data object Cancelled : PaymentError       // el usuario cancela el 3DS
    data class Unknown(val message: String) : PaymentError
}

Reintentos con criterio

Solo se reintentan errores transitorios (Network, Timeout, RateLimited), con backoff exponencial y límite. Nunca se reintenta un Declined o InvalidCard. Y el reintento reutiliza la misma IdempotencyKey, para que el PSP no cobre dos veces.

kotlinsuspend fun retryTransient(
    maxAttempts: Int = 3,
    initialDelayMs: Long = 300,
    isTransient: (PaymentError) -> Boolean,
    block: suspend () -> AuthResult,
): AuthResult {
    var delay = initialDelayMs
    repeat(maxAttempts - 1) {
        val r = block()
        if (r !is AuthResult.Error || !isTransient(r.error)) return r
        delay(delay); delay *= 2
    }
    return block() // último intento
}
El detalle que separa junior de seniorSaber qué NO se reintenta (un Declined jamás) y la idempotencia (que un timeout + reintento no genere doble cobro). Ten esos dos tests en verde y sabe explicarlos.

kotlin · commonTest@Test fun `rechazo emite Failed(Declined) y no reintenta`() = runTest {
    val repo = FakePaymentRepository(outcome = Declined)
    val states = ProcessPaymentUseCase(repo)(request).toList()
    assertEquals(PaymentState.Failed(PaymentError.Declined), states.last())
    assertEquals(1, repo.authorizeCalls)              // NO reintentó
}

@Test fun `reintento reutiliza la key y no cobra dos veces`() = runTest {
    val psp = FakePsp()
    val r1 = psp.authorize(request); val r2 = psp.authorize(request)  // misma key
    assertEquals(r1, r2)
    assertEquals(1, psp.charges)                      // un solo cargo
}

Roadmap por fases

Rama a rama (tu flujo). Cada fase trae su prompt para pedírselo a Claude Code. Empieza por el dominio: si clavas la máquina de estados y los casos de uso testeados, lo demás cae solo.

Setup KMP + repo

Proyecto KMP (:shared + :androidApp), version catalogs, dependencias, repo en GitHub y ramas.

Crea un proyecto Kotlin Multiplatform con :shared (androidTarget; iOS preparado pero opcional) y :androidApp con Compose. Version catalogs. commonMain en capas domain/ y data/, commonTest para tests. Añade Koin (KMP), kotlinx-coroutines, kotlinx-datetime y Turbine.

Dominio en commonMain

Lo difícil primero: modelos, máquina de estados y casos de uso, Kotlin puro.

En :shared/commonMain/domain implementa los modelos (Amount, Currency, PaymentMethod, CardToken, IdempotencyKey con kotlin.uuid.Uuid, PaymentRequest, Receipt, PaymentError), PaymentState y ProcessPaymentUseCase (con completeSca). Añade Luhn y caducidad con kotlinx-datetime. Kotlin puro, sin Android.

Tests del dominio

commonTest con Turbine: cubre caminos felices y de error.

En commonTest escribe tests de ProcessPaymentUseCase con un FakePaymentRepository que reproduzca Approved, NeedsSca, Declined y Error; tests de Luhn y de la máquina de estados. Usa Turbine.

Data (commonMain)

FakePsp con idempotencia + tokenizer PCI-safe.

En :shared/commonMain/data implementa PaymentRepository con un FakePsp configurable (approved/needsSca/declined/network) con latencia e idempotencia por IdempotencyKey, y un CardTokenizer PCI-safe (tokeniza y enmascara, nunca loguea el PAN). Tests en commonTest de idempotencia y enmascarado.

UI Android (Compose + MVI)

Selección de método y formulario de tarjeta con validación en vivo.

En :androidApp crea la pantalla de selección de método de pago y el formulario de tarjeta (validación en vivo con Luhn, formateo, enmascarado) con MVI (estado inmutable e intents), consumiendo los casos de uso de :shared. Design system y tema propios.

3D Secure

Challenge, OTP simulado y completeSca.

Añade la pantalla de challenge 3DS: cuando el estado sea RequiresSca, navega al challenge, pide un OTP simulado y llama a completeSca. Maneja éxito, ScaFailed y cancelación del usuario (Cancelled).

Accesibilidad

traversalIndex, liveRegion y key(...) — tus fixes de Santander.

Aplica accesibilidad al checkout: traversalIndex para el orden de foco, liveRegion para anunciar errores y el resultado del pago, contentDescription en marcas de tarjeta, y key(...) para evitar anuncios indebidos de placeholders que cambian.

Errores y resiliencia

La fase clave en pagos: taxonomía, retryTransient e idempotencia.

Implementa la taxonomía completa de PaymentError, un mapper PSP→PaymentError centralizado en el borde, y retryTransient con backoff que SOLO reintenta transitorios reutilizando la misma IdempotencyKey (nunca Declined/InvalidCard). Recibo (éxito) y pantalla de fallo con mensaje accesible por tipo de error y botón de reintento. 3DS: cancelación/timeout como Cancelled/Timeout. Añade los tests de la matriz, sobre todo los de error e idempotencia. Verifica que ningún log contiene el PAN.

README con el diseño, un diagrama de la máquina de estados y una sección "¿Qué demuestra este proyecto?" orientada a pagos. Repasa que ningún log/estado contiene el PAN completo.
